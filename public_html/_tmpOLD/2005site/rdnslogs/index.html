What is RDNSLOGS?

RDNSLOGS is program that reads the log files generated by web servers and produces a cache file of DNS Reverse Lookups that is immediately compatible with the Analog log file analyzer. The program converts the numeric IP addresses found in log files (or any text file for that matter) to their corresponding host names. RDNSLOGS will read the specified analog configuration file for LOGFILE specs. It will process these files whether they be compressed and uncompressed to find all IP addresses and translate them to their corresponding domain names. RDNSLOGS has internal support for gzip compression and has support for all external forms of log file compression. Through the use of multiple concurrent reverse DNS lookup threads, a DNS cache file compatible with Analog is produced. This multi-threaded lookup process is orders of magnitude faster than the single, sequential lookup feature built into Analog.

Many have expressed an interest in using RDNSLOGS with systems such as Webalizer. This is certainly possible since the cache file produced by RDNSLOGS is a simple text file whose format is:

timestamp IP_address name

where the timestamp is the number of minutes since the beginning of 1970, GMT (i.e., "Unix time" divided by 60), and the name is just * if the address couldn't be resolved. Knowing this format, it should be a simple matter of writing a PERL, AWK, or other text processing script that can transform the output of RDNSLOGS to whatever system you currently use.

How to Use RDNSLOGS

RDNSLOGS is free software licensed under the GNU Public License. The java source code is included for a nominal fee.

Note: The Java Runtime Environment (JRE) is required to use RDNSLOGS Java.

The command line syntax is as follows:

java RdnsLogs [options] <analog configuration filename>

Option Description Implemented
-t threads Max count of concurrent DNS lookups (thread count) (default: 20, maximum: 400) V1.0
-v Verbosity switch.  Use the -v switch only when you want to monitor the state of RDNSLOGS.  It generates copious amounts of output and generally slows the program. V1.0
-l logfile You specified a log file specification on the command line. i.e. -l c:\logs\*.gz You can specify as many logfile specs as can fit on a command line provided that each spec is preceded with a -l. If a log file spec contains a space, surround the spec with quotes. If a log file spec is made, an analog config file does not need to be specified. V1.0

Note: For optimal heap management use -Xms and -Xmx ( i.e. java -Xms 256 -Xmx 512 )

Command Line switches are processed before the config file, if a config file is specified.

Example 1:

C:> java RdnsLogs -t 100 analog.cfg

This command line will read the analog.cfg file in the current directory for DNS related information and use up to 100 threads to resolve IP addresses in log files.

Example 2:

C:> java RdnsLogs

Displays a complete usage summary including version number and available options.

Example 3:

C:> java RdnsLogs -v -t 150 myanalog.cfg

Read the file myanalog.cfg for log file locations and use up to 150 threads to resolve the IP addresses in those files. Also, generate debugging information on the screen.

Example 4:

C:> java RdnsLogs -l c:\logs\*.log -l "d:\log files\*.gz" -o mynewdns.txt -t 300

Reads all the *.log files in the C:\logs directory and all the *.gz files in the "d:\log files" directory to create a mynewdns.txt file in the current directory using 300 lookup threads.

You do not need to create another configuration file to use RDNSLOGS. RDNSLOGS will read your current analog configuration file for the following command lines:

  • LOGFILE lines specify the directory locations of your log files. You can have multiple LOGFILE commands.

  • RDNSLOGS will process each specified directory.

  • DNSFILE command specifies the name of your dns cache file.

  • DNSGOODHOURS specifies how many hours a successfully resolved IP address is considered valid.

  • DNSBADHOURS specifies how many hours an unsuccessfully resolved IP address needs to wait before attempting to resolve that IP address again.

  • TO specifies the final timestamp that will be considered in a log file analysis. It is used when the percent specifiers are used in log file names. If no TO command line is found, analog and RDNSLOGS assume the current time when these programs are executed.

  • UNCOMPRESS specifies the external program to call if you want to uncompress log files that are compressed using a format other than gzip.


All other lines in the configuration file are ignored.

Remember, there is nothing that says you need to tie RDNSLOGS with an existing Analog configuration file. You can create a separate text file containing the command lines discussed above and specify that file as the configuration file for RDNSLOGS.

If all goes well, a DNS cache file is created that contains IP addresses and their associated domain names. The file is used by analog for its own host information reporting. Remember to include the line DNS READ in your analog configuration file. While RDNSLOGS does not use that line at all in its processing, DNS READ is needed by ANALOG to process the DNS cache file.

Please see analog's extensive help files, http://www.analog.cx/docs/Readme.html, for more information on the specific command lines discussed here as well as a more comprehensive description of its use of the dns cache file.

It would be most advantageous to use the output produced by RDNSLOGS as a way to monitor its progress. For instance, the following command in the windows scheduler:

" java RdnsLogs -t 100 analog.cfg > progress.txt"

will cause output to be redirected into a file called progress.txt. You may periodically view the file to see the state of RDNSLOGS.
